## 2. השרשרת — הרעיון המרכזי

```
יומן גוגל
   ↓ (אוטומטי, כל בוקר 06:00)
הפקה נוצרת [ממתין_להתחלה]
   ↓
6 שלבים נוצרים אוטומטית:
   קו פרק:  הקלטה → עריכה → שליחה
   קו רילז: הקלטה → עריכה → שליחה
   ↓ (טכנאים מעדכנים)
כל 6 השלבים done → [נשלח_ללקוח]
   ↓ (אני/החשבונאית מסמנים)
[אושר_ע"י_לקוח]  ← זה הטריגר לחיוב
   ↓ (אוטומטי — רק אם kind='client')
job נוצר + חשבונית עסקה מונפקת במורנינג
   ↓
⏳ ממתין לתשלום  ← ערוצי ה-VU ברדאר
   ↓ (מסמנים "שולם")
🔴 "חשבונית מס נדרשת" — עד שהיא מונפקת
   ↓ (קליק אנושי, לעולם לא אוטומטי)
✅ סגור
```

### עקרון הבטיחות
**חשבונית עסקה** = דרישת תשלום, לא מסמך מס, הפיכה → **בטוחה לאוטומציה.**
**חשבונית מס** = מסמך משפטי, ביטולו דורש חשבונית זיכוי → **תמיד בקליק אנושי.**

זו הסיבה שהאוטומציה בטוחה: אם טכנאי גורר כרטיס בטעות, המקסימום שיצא הוא דרישת תשלום שאפשר למחוק. **לעולם לא מסמך מס.**

## 3. הרשאות — הדרישה הקריטית ביותר

### הבעיה במה שבנית
`role` קובע הכל. אבל בפועל **יש טכנאים שכן צריכים לראות כסף.**
`role` לא יכול להיות כלוב — הוא צריך להיות **קיצור דרך**.

### המבנה הנכון — הרשאות אטומיות פר-משתמש

```sql
profiles:
  id, name, email, role, approved boolean default false
  can_view_money      -- לראות jobs, סכומים, חוזים
  can_edit_money      -- לערוך paid, להנפיק חשבוניות
  can_view_stages     -- לראות את לוח ההפקות
  can_edit_stages     -- לעדכן סטטוס שלבים
  can_manage_users    -- אדמין: לאשר משתמשים ולתת הרשאות
  can_import          -- לייבא CSV
```

`role` = **פריסט בלבד.** לחיצה עליו מדליקה תיבות, ואפשר לשנות ידנית אחר כך:
- `owner` → הכל
- `tech` → `can_view_stages` + `can_edit_stages`
- `bookkeeper` → `can_view_money` + `can_edit_money` + `can_view_stages` (קריאה)

### RLS — נאכף במסד, לא ב-UI
```
jobs / contracts / invoices:
  SELECT  רק אם can_view_money
  UPDATE  רק אם can_edit_money
stages:
  SELECT  רק אם can_view_stages
  UPDATE  רק אם can_edit_stages
profiles:
  UPDATE  רק אם can_manage_users
```

### חוקי ברזל
- **אף אחד לא משנה את ההרשאות של עצמו.** גם לא אדמין.
- **`approved=false` → אפס גישה.** משתמש חדש רואה רק "ממתין לאישור".
- המשתמש `bnaya` = owner, approved, כל התיבות דלוקות. נעול כאדמין ראשון.

### מסך ניהול משתמשים (רק `can_manage_users`)
טבלה: שם | אימייל | תפקיד | ✅מאושר | 6 תיבות סימון | מחיקה
- משתמש נרשם → נכנס כ-`approved=false` → התראה בדשבורד שלי
- כפתורי פריסט מהירים: `[טכנאי]` `[חשבונאית]` `[אדמין]`

### מבחן קבלה — חובה, שני כיוונים
```
1. משתמש עם can_view_money=false → משיכת jobs דרך API → 0 שורות
2. הדלק לו can_view_money=true    → משיכה שוב      → 157 שורות
שני הכיוונים חייבים לעבוד. אם לא — עצור ותקן.
```

## 4. מודל הנתונים

### `productions` — יחידת העבודה הבסיסית
```sql
id, podcast_name, client_id (nullable), contract_id (nullable),
guest, record_date, studio, calendar_uid,
kind enum ('client' | 'internal'),
status enum ('ממתין_להתחלה'|'הוקלט'|'בעריכה'|'נערך'|'נשלח_ללקוח'|'אושר_ע"י_לקוח'),
studio_hours numeric, edit_hours numeric,

```

**חוק ברזל:** רק `kind='client'` יוצר `job` או חשבונית. לעולם.
**חסימה:** `kind='client'` בלי `client_id` → לא ניתן לשמור.

**זריעה:** כל 709 ההפקות → `kind='internal'` כברירת מחדל.
אני אסמן ידנית. **תן לי bulk edit** — לא אסמן 709 שורות אחת אחת.

### `stages` — 6 לכל הפקה
```sql
id, production_id,
track enum ('episode' | 'reels'),
step  enum ('record' | 'edit' | 'deliver'),
status enum ('pending' | 'in_progress' | 'done'),
assignee_id, done_at
```
- הפקה חדשה → **6 שלבים אוטומטית** (טריגר), כולם `pending`
- שלב נפתח רק אחרי שקודמו `done`
- `productions.status` **נגזר מהשלבים** — לא מוזן ידנית

### `clients`
```sql
id, name, normalized_name, contact_name,
billing_mode enum ('per_episode'|'retainer'|'package'|'none') default 'per_episode',
payment_terms enum ('immediate'|'net_30'|'net_60'|'eom_30'|'eom_60'|'eom_90') default 'immediate',

```

### `contracts` + `contract_milestones` — חוזים גדולים
```sql
contracts:
  id, client_id, name, total_amount, start_date, end_date,
  status enum ('active'|'closed')

contract_milestones:
  id, contract_id, name, amount,
  expected_date date, is_estimated boolean,
  status enum ('pending'|'invoiced'|'paid'),
  job_id (nullable)
```

### `jobs` — שורת חיוב
```sql
id, production_id (nullable), client_id, contract_id (nullable),

paid enum ('כן'|'לא'|'ללא חיוב'|'לא ידוע'),

```

### `invoices` — טבלה חדשה (`invoice_biz`/`invoice_tax` הם שריד מהאקסל)
```sql

morning_doc_id text UNIQUE,   -- ⚠️ UNIQUE! לעולם לא להנפיק פעמיים

```
**חשבונית אחת יכולה לכסות כמה jobs** (ריטיינר חודשי = 4 פרקים, חשבונית אחת).
העבר את הנתונים מהעמודות הישנות לטבלה החדשה.

### `events` — טבלת אירועים · הבסיס להרחבה
```sql
id, entity_type, entity_id, event_type, actor_id, payload jsonb, created_at
```
כל שינוי סטטוס, כל חשבונית, כל תשלום — **נרשם כאן.**
זה מה שיאפשר להוסיף מודולים בעתיד בלי לגעת בליבה.

## 5. חישוב חוב — המנוע הנוכחי שגוי

### הבעיה
חישבנו גיל חוב **מתאריך העבודה.** אבל ידיעות אחרונות הם **שוטף+60**.
עבודה מ-1.5 → חשבונית 31.5 → **פירעון 31.7.**
היא נראית "60-90 יום, כתום" — **אבל היא בזמן לחלוטין.**

**התראות שווא = הצוות מפסיק להאמין לרדאר = המערכת מתה.**

### התיקון
```
due_date מחושב לפי clients.payment_terms:
  immediate → תאריך החשבונית
  net_30    → + 30 יום
  net_60    → + 60 יום
  eom_30    → סוף החודש + 30
  eom_60    → סוף החודש + 60   ← ידיעות אחרונות
  eom_90    → סוף החודש + 90

אם טרם הונפקה חשבונית → חשב לפי תאריך העבודה + סמן "משוער".
```

### 4 ערוצי ה-VU — ספירה מ-`due_date`, לא מתאריך העבודה
```
✅ ירוק  = טרם הגיע מועד הפירעון   ← "בזמן", לא חוב!
🟡 1-30 יום באיחור
🟡 31-60 יום באיחור
🔴 60+ יום באיחור
```
**שינוי משמעותי:** חוב שנראה אדום היום עשוי להיות ירוק. **זה נכון ורצוי.**

### שני מספרים נפרדים — לעולם אל תערבב
| | |
|---|---|
| **חוב לגבייה** | חויב, לא שולם → 39,500 ₪ |
| **התחייבות פתוחה** | טרם חויב → 250,000 ₪ |

הדשבורד מציג את **שניהם, בנפרד, ברור.**
